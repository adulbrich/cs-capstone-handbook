Team Submission

Purpose

  • Align your team on a shared system vision before deep implementation.
  • Make interfaces and contracts explicit to reduce integration risk.
  • Surface and prioritize project risks early with concrete mitigations.
  • Practice owning a component by writing a concise, evidence-backed design.

What You Must Produce

Submit one technical design brief (PDF, ~3–8 pages with core content, more if required by your project partner + appendix).

Required Report Structure

Your submission must include the following sections. 

0. Change Log

Briefly record how this document has changed since the previous version. Focus on high-level edits (which sections were updated and why), not on restating every design decisions.

For each revision, include:

  • Date of the update
  • Version or Tag (e.g., v1.1, “Winter Week 4 Update”)
  • Sections Changed (“Interfaces/APIs and Data Contracts”, “Component Design and Responsibilities”, etc.)
  • What Changed (1–2 sentence summary)
  • Reason / Driver (e.g., stakeholder feedback, implementation discovery, new risk discovered)
  • Author(s) — individual names or subteam

A table format is recommended:

Date Version Sections Changed Summary of Changes Reason / Driver Author(s)
2026-02-15 v1.1 Interfaces/APIs and Data Contracts; Component Design and Responsibilities Refined API contracts and updated component responsibilities to match the current implementation plan. Implementation spike results and updated sponsor feedback. J. Doe
2026-03-01 v1.2 Technical and Integration Risk Register; Design Decisions and ADR Links Re-ranked technical and integration risks; added new ADRs for data model changes and third-party service integration. New performance findings and integration constraints discovered during Winter Sprint 4. Everyone

Note: The Change Log is a brief history of document updates, not a restatement of all design decisions.

1. System Context and High-Level Architecture

Describe how your system fits into its environment and how major pieces fit together.

  • Identify external actors (users, services, devices, third-party APIs).
  • Show major containers/components and how they interact (e.g., C4 level 1–2 view).
  • Name connectors and protocols (HTTP/REST, gRPC, queues, database connections, etc.).
  • State boundaries and key deployment assumptions (e.g., cloud/on-prem, major services, data stores).
  • Include at least one architecture diagram with a figure number, caption, and alt text.
  • Ensure the diagram is consistent with your current or planned repo structure.

2. Interfaces/APIs and Data Contracts

Make the key interfaces and contracts explicit.

  • List the key interfaces/APIs and which components own them.
  • Provide contracts (schemas/specs) for important requests, responses, and messages (e.g., OpenAPI, Protobuf, JSON Schema, or clear typed tables).
  • Include example requests/responses for key endpoints or messages.
  • Describe expected error cases and how errors are represented.
  • Summarize your approach to authentication, authorization, and versioning (paths, headers, tokens, feature flags, etc.).
  • Reference any attached schema/spec files from this section.

3. Component Design and Responsibilities

Explain how the system is decomposed into components and what each is responsible for.

  • Define the main components or services in your system and their responsibilities (what each owns and does not own).
  • Show how components relate to the interfaces/APIs defined in Section 2 (who calls what, who owns which contracts).
  • Describe key dependencies between components and external services (databases, queues, third-party APIs, libraries).
  • For each component, note any important performance, scalability, or operational considerations (e.g., expected load, latency sensitivity, scaling strategy).
  • For each component, identify 1–2 component-specific technical or integration risks (e.g., resource hotspots, fragile integrations, data model churn) and link them to entries in the shared Technical & Integration Risk Register in Section 4 (by risk ID, e.g., R-03).

4. Technical and Integration Risk Register

Capture the most important technical and integration risks and what you will do about them.

  • Provide a risk register table with at least the following columns: ID, Risk, Impact, Likelihood, Mitigation / Acceptance
  • Focus on implementation and architecture risks, such as performance and scalability, integration with third-party services, tech stack immaturity, data model churn, security/ops, observability, deployment, and rollback risks.
  • Do not re-list pure requirements or stakeholder risks. Instead, link to the relevant requirements or stakeholder documents where needed.
  • Include and re-rank the technical and integration risks contributed by your components (from Section 3) so the register reflects a shared, prioritized view of system-level technical risk.
  • Rank or otherwise indicate relative priority based on impact and likelihood.
  • For each risk, give a concise mitigation or acceptance plan.
  • Where relevant, reference requirements, components, or ADRs that are shaped by this risk.

5. Design Decisions and ADR Links

Show how you document and adopt key design decisions.

  • Reference any Architecture Decision Records (ADRs) or decision notes relevant to your component and system.
  • For each major decision, briefly summarize:
    • Context and options considered.
    • The decision and primary tradeoffs.
    • Important consequences (good and bad).
  • Include ADR status (e.g., proposed, accepted, superseded) where applicable.
  • Link to PRs or issues where the decision has been adopted or implemented, if available.

Formatting Guidelines

  • Keep the main team sections to ~3–8 pages. Use appendix for supporting information.
  • Use clear headings, subheadings, bullets, and descriptive link text.
  • Write for a technical stakeholder: be concise, define project-specific terms on first use, and keep terminology consistent across the document.
  • Use consistent notation and naming between diagrams, text, and code.
  • Give all figures figure numbers, captions, and alt text.
  • Explain non-obvious notation.
  • Ensure diagrams and text are readable when printed or viewed on a laptop screen.
  • Ensure a reader can quickly answer: “Which part of the system addresses this risk?” and “Where is that decision documented?”
  • Generative AI may be used for writing. Do not fabricate artifacts, evidence, or data; all examples, metrics, and links must be real.
