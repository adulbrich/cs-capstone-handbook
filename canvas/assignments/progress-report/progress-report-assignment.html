Team Submission with Individual Contributions

Purpose

  • Report measurable, verifiable sprint progress rather than intentions.
  • Demonstrate traceability between requirements → tasks → PRs/issues → (deployable) software.
  • Maintain consistent risk and quality tracking across sprints.
  • Practice concise, professional technical communication.
  • Strengthen accountability through transparent documentation of individual contributions.

What You Must Produce

Submit one team progress report (PDF, 1-2 pages). Individual grades depend on demonstrated contributions. There is a separate assignment for graders to deduct points at the individual level. By default, you will get max points on that assignment.

  • Filename format: YYYY-MM-DD-[TeamID]-Sprint6-ProgressReport.pdf
  • Example: 2026-01-18-CS.042-Sprint6-ProgressReport.pdf
  • Template: TBD

Required Report Structure

Your submission must include the following sections. Each section aligns to one or more rubric criteria.

1. Summary

A concise overview of the sprint: major outcomes, blockers, and key insights (3-5 bullets).

2. Sprint Work Log (Work Items + Evidence + Attribution)

List the work items you completed or meaningfully worked on during the sprint. A list format is preferred over a table (tables can become hard to read with many items).

Each item must be measurable and include evidence and attribution. Items can be requirements work, bug fixes, infrastructure work, stakeholder/marketing/outreach tasks, research/spikes, documentation, etc.

For each work item, include:

  • Title (short, specific)
  • Status (done / partial / not done) or a clear %
  • Owner / Author (who did the primary work)
  • Reviewer(s) / Collaborator(s) (if applicable)
  • Source link (issue, task, PR, sprint plan item, requirement, test, ADR, or doumentation, i.e., something trackable)
  • Supporting artifact (at least one verifiable artifact)
    • Deploy URL, screenshots, demo clip
    • CI/build status or brief test summary
    • Write-up artifacts (ADR, design doc, spike results, interview summary, decision log)
    • Diagrams/specifications
  • Notes (optional): blockers, plan changes, or what remains

One artifact may cover multiple work items (e.g., one demo clip shows several features), as long as you clearly say which items it covers.

If an artifact corresponds to a code state, include the relevant commit hash(es). If an artifact is a write-up, include a stable link and enough context to evaluate it (title/date/version/location).

3. Risk and Quality Tracking

List the number of recorded bugs currently open and the highest-impact open bugs.

  • Bug count (1 line): Open bugs now = [N]. Include a link to your bug list / issue query.
  • Top bugs (3–5 items): list your highest-impact open bugs.
  • Each bug must include: severity/impact, owner, current status, and a link.
  • Actions: assign a concrete next step for the top 2–3 bugs (who does what by when).

If you truly have no bugs: write “No known bugs” and link to where you track issues, plus 1 sentence describing what you did to check (e.g., manual test pass, demo run-through, CI green).

4. Next Sprint Plan

List the most important next sprint work items (3-5 items). These do not have to be requirements—they can be bug fixes, infrastructure work, stakeholder/marketing/outreach tasks, research/spikes, documentation, or anything else that produces a measurable outcome.

Each item must include a link to a verifiable source of truth, such as a requirement ID, test ID, GitHub issue/task, ADR/design doc, marketing plan doc, or other trackable artifact. This supports traceability and makes planning gradeable.

5. Team Process Reflection

Write 3-5 sentences reflecting on:

  • What worked
  • What didn’t
  • One or two actionable improvements
  • What evidence will show the improvements worked

Formatting Guidelines

  • Keep the main body around 1-2 pages, with screenshots/clips linked or appended.
  • Use clear headings, bullets, and descriptive link text.
  • All images require alt text.
  • Do not attach binaries; use links to repo/CI/deploy.
  • Generative AI may be used for writing. Do NOT fabricate artifacts or data.
