Team Submission with Individual Contributions

Purpose

  • Report measurable, verifiable sprint progress rather than intentions.
  • Demonstrate traceability between requirements → tasks → PRs/issues → (deployable) software.
  • Maintain consistent risk and quality tracking across sprints.
  • Practice concise, professional technical communication.
  • Strengthen accountability through transparent documentation of individual contributions.

What You Must Produce

Submit one team progress report (PDF, 1-2 pages). Individual grades depend on demonstrated contributions.

  • Filename format: YYYY-MM-DD-[TeamID]-Sprint6-ProgressReport.pdf
  • Example: 2026-01-18-CS.042-Sprint6-ProgressReport.pdf
  • Template: TBD

Required Report Structure

Your submission must include the following sections. Each section corresponds directly to a rubric criterion.

1. Summary

A concise overview of the sprint: major outcomes, blockers, and key insights (3-5 bullets).

2. Progress vs. Plan 

A table with the following columns:

  • Planned Item
  • Status (% or done/partial/not done)
  • Links (issue/PR)
  • Notes and Blockers
  • Plan Change (if any)

You can add the author/reviewers for each entry as well (see section 7 about individual contributions).

3. Supporting Artifacts (Evidence)

Provide at least one verifiable supporting artifact for each item in Section 2: Progress vs. Plan (or one artifact that clearly covers multiple items). Artifacts may show working software or other measurable progress (e.g., research or design work).

Supporting artifacts can be any of:

  • Deploy URL
  • 1–3 screenshots
  • ≤2-minute demo clip 
  • CI/build status (badge or link)
  • Brief test summary (e.g., total tests, failures, coverage ±Δ, manual/qualitative)
  • Write-up artifacts (e.g., ADR, design doc, spike results, interview summary, decision log)
  • Diagrams or specifications
  • Any other artifact that a grader can verify

If an artifact corresponds to a code state, indicate the relevant commit hash(es). If an artifact is a write-up, include a stable link and enough context to evaluate it (title/date/version/location).

Example: a single deployment link or demo clip can cover multiple completed items, as long as it clearly demonstrates them.

4. Risk & Quality Tracking

  • Top 2–3 active risks, each with:
    • owner
    • current status
  • Bug and test metrics and trends: counts, link to relevant issues/tasks, severity if relevant, failures, flaky tests, missing coverage, etc. 
  • Concrete actions assigned to specific owner(s)

5. Next Goals

Each goal must link to a requirement ID, test ID, or corresponding issue/task (3-5 items). This ensures traceability and supports planning for the next sprint.

6. Team Process Reflection

Write 3-5 sentences reflecting on:

  • What worked
  • What didn’t
  • One or two actionable improvements
  • What evidence will show the improvements worked

7. Individual Contributions

For every teammate, list:

  • PRs authored
  • PR reviews
  • Issues created/closed
  • Other measurable contributions (documentation, research, interviews, administrative overhead, etc.)

Each item must include links. Measurable means that we can see the results: "I worked on researching a database schema" is not measurable, provide instead a proper write-up of the research, decisions, and final artifacts and save it in a place it can be referenced to (e.g., ADRs, issues, wiki, user story or acceptance criteria, Figma design, etc.)

Formatting Guidelines

  • Keep the main body around 1-2 pages, with screenshots/clips linked or appended.
  • Use clear headings, bullets, and descriptive link text.
  • All images require alt text.
  • Do not attach binaries; use links to repo/CI/deploy.
  • Generative AI may be used for writing. Do NOT fabricate artifacts or data.
