Team Submission

Purpose

  • Give your project partner everything they need to continue, maintain, or extend the project without your involvement.
  • Leave a complete record of all work your team produced across all three terms.
  • Practice professional knowledge transfer, a real skill in any software engineering role.

What You Must Produce

Submit two things by the deadline:

  1. A partner handoff document (PDF, 2–4 pages), submitted to Canvas
  2. Your course archive (all project artifacts from all three terms), submitted via the drop link in this assignment
  • Handoff document filename: YYYY-MM-DD-[TeamID]-ProjectHandoff.pdf
  • Example: 2026-06-08-CS.042-ProjectHandoff.pdf

Student-driven projects: If you have no external project partner, have your mentor or instructor serve as the client for partner verification.

Part 1: Partner Handoff Document

This document is for your project partner (or a future developer picking up the project). It should be professional, concise, and actionable. A new person with relevant technical background should be able to run and maintain the project using only this document and the repository.

1. Project Overview and Status

A short summary (half a page or less):

  • What the project is and what problem it solves
  • Current deployment or delivery state: is it running? Where? How stable?
  • What is and is not included in this handoff (any exclusions and why)
  • Contact information for each team member (for follow-up questions)

2. Setup and Getting Started

Step-by-step instructions for getting the project running from scratch. Write this for someone who has never seen the codebase. Include:

  • Prerequisites (OS, runtime versions, tools, accounts, credentials)
  • Installation and build steps (copy-pasteable commands)
  • How to run the test suite
  • How to run a local development environment
  • How to deploy (or a pointer to the deployment runbook)
  • Common pitfalls and how to resolve them

3. Known Issues, Limitations, and Future Work

An honest, prioritized accounting of the project’s current state:

  • Known bugs: link to open issue tracker (or list them if no tracker); include severity
  • Limitations: features that are incomplete, descoped, or working but fragile
  • Technical debt: areas that need refactoring or re-architecture
  • Future work: prioritized backlog of features or improvements not yet implemented
  • Any credentials, API keys, environment secrets, or access permissions that need to be transferred

4. Reference Documents

Include links to (or copies of) the final versions of your requirements document and technical design document. These give your partner the context they need to understand what was built, what was descoped, and the rationale behind key design decisions.

5. Handoff Checklist and Partner Verification

Complete this checklist and include it in your document. Check each item that is complete. For any unchecked item, include a brief note explaining why and what follow-up is needed.

  • [ ] Repository access transferred to project partner (or archived)
  • [ ] All credentials, API keys, and secrets shared securely (not committed to the repo)
  • [ ] Deployment environment access transferred or documented
  • [ ] All open issues triaged and documented
  • [ ] README is current and accurate
  • [ ] Final requirements document and technical design document linked or attached
  • [ ] Partner (or substitute) has confirmed receipt in writing

The last item requires written confirmation from your project partner (or instructor/mentor for student-driven projects). Include this confirmation directly in the document: a quoted email, a signed statement, or equivalent written evidence that the partner can access the repository, run the project, and understands the maintenance requirements going forward.

Part 2: Course Archive

The course archive is a complete record of everything your team produced across all three terms. It is for the course, not the project partner.

Use the drop link provided in this assignment to submit your archive. Include everything: the partner handoff document, requirements documents, design documents, ADRs, team charter, sprint progress reports, test plans, retrospectives, meeting notes, presentations, demo videos, code exports, and any other artifact your team produced. Do not curate or filter; if it was produced for this course, include it.

Also include a short release consent declaration (a plain text or PDF file) at the top level of your archive. The OSU School of EECS may share project archives with future capstone students, prospective students, or sponsoring companies as examples of student work. This is entirely optional and has nothing to do with your project partner’s rights to use the software you built. List every team member and mark each as Consenting or Not Consenting. If any team member declines, or if your project has an NDA or any other restriction on external sharing, the archive will not be shared, but you still submit it to the course regardless.

Formatting Guidelines

  • Handoff document (Sections 1–5): 2–4 pages; clear headings and bullets throughout. The handoff checklist does not count toward the page limit.
  • Code blocks for all commands and file paths.
  • All links must be descriptive and working (or shared with the instructor and partner).
  • Include alt text and captions for any screenshots or diagrams.
  • Generative AI may be used for writing. Do NOT fabricate steps, links, or artifact locations.
