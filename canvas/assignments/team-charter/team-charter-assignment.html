Team Submission

Purpose

  • Align team expectations early so you can move fast without stepping on each other.
  • Practice professional communication and ethical team norms you’ll use in industry.
  • Define quality bars (Definition of Done) that keep your code shippable.
  • Document inclusive practices so everyone can contribute effectively.

What You Must Produce

Submit your Team Charter and Working Agreement document (PDF) and Contribution Guidelines (Markdown)

Required Team Charter Structure

Create a professional team policy document that defines how you will work together. This document must include a change log and the following policy sections.

0. Change Log

Briefly record how this document has changed since the previous version. Focus on high-level edits (which sections were updated and why).

For each revision, include:

  • Date of the update
  • Version or Tag (e.g., v1.0, “Fall Charter”)
  • Sections Changed
  • What Changed (1–2 sentence summary)
  • Reason / Driver (e.g., team retrospective, new tool adoption)
  • Author(s) — individual names or subteam

A table format is recommended:

Date Version Sections Changed Summary of Changes Reason / Driver Author(s)
2026-02-15 v1.1 Meeting Cadence & Tools Updated meeting times for Winter term and added Zoom link for remote sessions. Schedule changes for new term. J. Doe
2026-03-01 v1.2 Definition of Done Added requirement for accessibility audit on frontend PRs. Team retrospective decision to improve a11y. Everyone

Note: The Change Log is a brief history of document updates, not a restatement of all policy edits. Detailed content belongs in the sections below.

1. Roles and Decision Making

Specify team member roles (including leadership/rotation) and responsibilities. Define your decision-making model (e.g., consent, majority, unanimity) for different types of choices.

2. Communication and Meetings

List your meeting cadence (times/days), communication channels, and core tools. Link to your meeting agenda/notes template.

3. Definition of Done (DoD) and Quality Gates

Define your quality bar. List required checks (tests, code review, security/lint/static analysis, docs updates) and reference the exact CI/CD gates that enforce them.

Point to your CI configuration, linter/formatter config, etc.

4. Conflict Resolution and Accountability

Write a restorative, stepwise plan for resolving conflict. Define how you will use evidence (PRs, reviews, attendance) to hold each other accountable.

5. Accessibility and Inclusion

Describe meeting norms (time zones, note-taking, turn-taking), how to request accommodations, and how you will surface and address barriers.

6. Risk Management

Identify key risks to the team process and define an escalation path for issues.

Required Contribution Guidelines Structure

Create a CONTRIBUTING.md file that lives in your repository. This document should include:

  • Prerequisites and local setup steps (how to clone, install dependencies, and run the project).
  • Commands for running CI, linters, and formatters locally.
  • The expected contribution workflow (branch naming, how to open a PR, and what information to include). Reference your Definition of Done (DoD). Describe code review expectations.
  • How to report bugs / request changes (where issues live and what details to include).
  • Where to ask for help (the team channel or point of contact).

Formatting Guidelines

  • Use consistent headings, list actionable steps, and assign an owner and next review date per section.
  • Keep links relative where possible; ensure public visibility for graders if using shared docs.
  • Keep the charter concise (aim for ~2–4 pages of focused, scannable content).
  • Generative AI may be used for writing. Do not fabricate artifacts, evidence, or data; all examples, metrics, and links must be real.
