Team Submission with Individual Contributions

This is a group submission that results in individual grades. Each team member must demonstrate specific, measurable contributions to receive credit. TAs and instructors reserve the right to take away up to 30 points per progress report for students who do not contribute meaningfully to the project.

Purpose

  • Stand up the minimum rails so the team can collaborate smoothly starting Sprint 2.
  • Agree on communication channels and cadences.
  • Create a tiny set of next tasks that align with near-term Canvas assignments and partner meetings.
  • If inheriting a repo (legacy code): capture a brief “current-state” summary and verify you can run it.

Task

Deliverables, Audience and Genre: Brief pre-sprint enablement memo for course staff and your partner.

Recommended Tools: GitHub (repo, issues, PRs Actions), optionally Linear or GitHub Projects (board). These are recommendations. You can use whatever tools you want.

  1. Repo and Basics (GitHub recommended)
    • New project: create team repo (empty/skeleton is fine)
    • Inherited project: ensure access to existing repo (or fork/mirror to your org)
    • Add/confirm: README (one-liner project description, team roster & contacts), .gitignore, and if known, LICENSE (confirm with project partner)
    • Invite all teammates + partner (read)
  2. PM Rails (GitHub Projects or Linear recommended)
    • Columns: Backlog → Ready → In Progress → Review → Done (or similar)
    • Inherited project: check existing backlog for tasks, discuss with project partner what to do with them
    • Add 5–10 light tasks (see “Next Tasks and Plan” below) and put due dates to match Canvas.
  3. Communication Mediums and Cadences
    • Choose primary async channel (Slack/Discord/Teams) and escalation path (email).
    • Add: standup cadence (e.g., Mon/Thu 10 min), weekly partner sync window, and response-time expectations (e.g., PR review ≤24h).
    • Document all of the above and pin in your chat.
  4. Branching and Reviews
    • Default branch set; agree on simple flow: feature/* → PR → ≥1 review → merge
    • Inherited project: check contribution guidelines
  5. (Optional) CI (GitHub Actions recommended)
    • Add a minimal workflow file (even a no-op “echo CI OK” is fine) or defer with a Linear task due early Sprint 2
  6. Inherited projects only:
    • Repo Summary (≤½ page): table with Languages/Frameworks, Build/Run commands (as you observed), External services (DB/APIs), Tests present (Y/N), Last release/tag, Open issues (#), Default branch.
    • Run Verification: copy the exact command(s) you used to run locally (or in a container), and state the observed result (e.g., “server started on :3000”, “CLI printed help”). A small screenshot or short log excerpt is fine.
    • Gap Notes (bullets): anything that blocked running (missing secrets, OS deps, tool versions), plus 2–3 small tasks you added to your backlog to close those gaps (e.g., “Document env vars (dev)”, “Add sample .env.local”, “Dockerize later—optional”).
    • Access and Permissions Check: confirm who can push/PR; partner/staff visibility set.
  7. Next Tasks and Plan
    • Populate your backlog with enough items for Sprint 2 (or later)
    • You can use tasks/issues to help split the work on upcoming assignments, document research, progress, and make sure everyone has their work defined
    • Check upcoming assignments + start planning for coding tasks

Deliverables and Format

Submit one report in PDF or Markdown:

  • Filename: YYYY-MM-DD-[TeamID]-Sprint1-ProgressReport.pdf/md (e.g, 2025-10-09-CS.042-Sprint1-ProgressReport.pdf)
  • Contents:
    • Executive Summary (4–6 bullets; ≤120 words): repo created, comms chosen, board live, 5–10 tasks queued, partner sync scheduled, and (if inherited) run verified / gaps noted.
    • Communication: table with channels, usage, owners, response times, recurring meetings (with days/times).
    • Repo Snapshot: screenshot of project repository and README, PR template if available.
    • PM Rails Snapshot: screenshot of project board showing at least 5 items.
    • (Optional) CI Snapshot: screenshot of CI setup and succesfully run even if empty.
    • Inherited projects: Repo Summary table + short Run Verification (command(s) and brief evidence).

Quality Checklist

  • Single, pinned async channel with documented meeting cadence and response-time expectations.
  • Repo and project board exist; a small, dated plan (5–10 tasks) is visible; partner and teammates have access.
  • New projects: PR template present; CI stub exists or a ticket is filed to add it early in Sprint 2.
  • Inherited projects: current-state snapshot captured; exact run command(s) and observed outcome recorded; follow-up gap tasks added.
  • (Optional) CI runs on PRs, or there is a near-term task scheduled to enable it.
